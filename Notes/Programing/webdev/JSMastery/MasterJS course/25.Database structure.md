###### Storing the ID of the data form other collection
#### When to use:
- Many-to-many relationships :When a document type can relate to multiple instances of another type and vice versa
- Frequent queries on related data 
- Light weight relationship: when the related documents are not overly complex and don't require a separate structure.
```TS
// students
{
	_id:"s1",
	name:"John",
	enrolledCourse:["c1","c2"]
}
// course
{
	_id:"c1",
	title:"Math 101",
	studentEnrolled:["s1","s2"]
}
```
Pros
- simplifies data retrieval, can fetch related documents with a single query if they are stored together
- easier to maintain
cons
- can lead to large documents if arrays grow significantly(not scale-able)
- requires extra queries to resolve reference, which may affect performance.
###### Solve this by creating multiple collections

---
### Creating multiple collections
###### When to use
- Complex relationship: when the relationships between documents are complex or when the entities are independent enough to warrant their collection.
- Side-load Data:If the related data has a distinct life-cycle or is managed independently
- flexibility and scalability:when you want to allow for growth in either collection independently without impacting the other.
```TS
// student collections
{
	_id:"s1",
	name:"John"
}
//course collections
{
	_id:"c1",
	title:"Math 101"	
}
// enrollement collections
{
	_id:"enroll1",
	studentID:"s1",
	corurseID:"c1",
}
```
###### pros
- Clean,clear separation of concerns
- facilitates better scalability independently
- reduces the document size
- fast
con
- more complex query need more logical steps
- adds overhead to maintain and ensure consistency across multiple collection

===> use multiple collection for big projects
